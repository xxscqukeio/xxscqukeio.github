URI 的格式

URI 本质 字符串

唯一 标记资源的 位置或者名字

不仅能够标记万维网的资源，也可以标记其他的，如邮件系统、本地文件系统等任意资源

磁盘上的静态文本 也可以说是 动态服务

由 scheme、host:port、path 和 query

scheme “方案名”或者“协议名” http、https

“://”之后，称为“authority”的部分 表示资源所在的主机名

“host:port” 主机名加端口号 主机名可以是 IP 地址或者域名的形式

使用默认的端口号 端口号有时可以省略

http 80

https 443

path 标记资源所在位置

URI 的 path 部分必须以“/”开始

HTTP 或 HTTPS 这样的网络通信协议，主机名是绝对不能省略

使用“协议名 + 主机名 + 路径”的方式

“query”部分 ?”表示开始 查询”的含义

多个“key=value”的字符串 字符“&”连接 长串的查询参数解析

URI 的完整格式

身份信息“user:passwd@”–登录主机时的用户名和密码–明文形式

查询参数后的片段标识符“#fragment”–“锚点”或者说是“标签”–获取资源后直接跳转到它指示的位置

片段标识符仅能由浏览器这样的客户端使用 服务器是看不到的

URI 转义的规则

把非 ASCII 码或特殊字符转换成十六进制字节值，然后前面再加上一个“%”

“银河”会被转义成“%E9%93%B6%E6%B2%B3”

支持任意的字符集用任何语言来标记资源

小结

URI 是用来唯一标记服务器上资源的一个字符串，通常也称为 URL；

URI 通常由 scheme、host:port、path 和 query 四个部分组成，有的可以省略；

scheme 叫“方案名”或者“协议名”，表示资源应该使用哪种协议来访问；

“host:port”表示资源所在的主机名和端口号；

path 标记资源所在的位置；

query 表示对资源附加的额外要求；

在 URI 里对“@&/”等特殊字符和汉字必须要做编码，否则服务器收到 HTTP 报文后会无法正确处理。

响应状态码该怎么用

状态码是三位数 从 000 到 999

第一位表示分类 000999 变成了 100599

1××：提示信息，表示目前是协议处理的中间状态，还需要后续的操作；

服务器也同意变更协议，就会发送状态码 101

2××：成功，报文已经收到并被正确处理；

服务器收到并成功处理了客户端的请求

3××：重定向，资源位置发生变动，需要客户端重新发送请求；

“301 Moved Permanently”俗称“永久重定向 已经不存在

“302 Found”，曾经的描述短语是“Moved Temporarily”，俗称“临时重定向

资源还在

4××：客户端错误，请求报文有误，服务器无法处理；

错误码

5××：服务器错误，服务器在处理请求时内部发生了错误。

服务器端的“错误码”

状态码在响应报文里表示了服务器对请求的处理结果；

状态码后的原因短语是简单的文字描述，可以自定义；

状态码是十进制的三位数，分为五类

从 100 到 599；2××类状态码表示成功

常用的有 200、204、206；3××类状态码表示重定向

常用的有 301、302、304；4××类状态码表示客户端错误

常用的有 400、403、404；5××类状态码表示服务器错误

常用的有 500、501、502、503。

你在开发 HTTP 客户端，收到了一个非标准的状态码，比如 4××、5××，应当如何应对呢？

500是php代码写的很烂，导致nginx的upstream接受错误，抛异常

404大多是浏览器请求facon.ico导致的 现在前后端分离根本没这个东西
