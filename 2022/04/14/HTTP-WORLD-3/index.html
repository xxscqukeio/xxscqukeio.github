TCP/IP 协议是一个“有层次的协议栈”

TCP/IP 网络分层模型

应用层 HTTP 消息或报文（message）

运输层 TCP/UDP 段（segment）

网际层 IP 包（packet）

链接层 MAC MAC 层的传输单位是帧（frame）

统称为数据包

层次顺序是“从下往上”

“链接层”（link layer）

在以太网、WiFi 这样的底层网络

发送原始数据包

MAC 地址来标记网络上的设备 MAC 层

“网际层”或者“网络互连层”（internet layer）

IP 地址取代 MAC 地址，

把许许多多的局域网、广域网连接成一个虚拟的巨大网络

找设备 IP 地址再“翻译”成 MAC 地址

“传输层”（transport layer）

保证数据在 IP 地址标记的两点之间“可靠”地传输

是 TCP 协议工作的层次 TCP/UDP

有状态的协议 先与对方建立连接然后才能发送数据

保证数据不丢失****不重复

UDP 则比较简单，它无状态

不用事先建立连接就可以任意发送数据

但不保证数据一定会发到对方

区别在于数据的形式

TCP 的数据是连续的“字节流”，有先后顺序

UDP 则是分散的小数据包，是顺序发，乱序收。

最经典的“三次握手”和“四次挥手”

协议栈的第四层叫“应用层”（application layer）

“百花齐放”了，有各种面向具体应用的协议

OSI 网络分层模型（Open System Interconnection Reference Model）

第一层：物理层，网络的物理形式，例如电缆、光纤、网卡、集线器等等；

第五层：会话层，维护网络中的连接状态，即保持会话和同步；

第六层：表示层，把数据转换为合适、可理解的语法和语义；

Layer seven”与“application layer”

TCP/IP 实际应用时的会话管理、编码转换、压缩等和具体应用经常联系的很紧密，很难分开

“四层负载均衡”

工作在传输层

基于 TCP/IP 协议的特性，例如 IP 地址、端口号等实现对后端服务器的负载均衡。

“七层负载均衡”

工作在应用层上

看到的是 HTTP 协议，解析 HTTP 报文里的 URI、主机名、资源类型等数据，再用适当的策略转发给后端服务器。

TCP/IP 协议栈的工作方式

HTTP 专用附加数据—>TCP 头—>IP 头、MAC 头—>P 层、MAC 层传输后拆包—>http

通过协议栈逐层向下，每一层都添加本层的专有数据，层层打包，然后通过下层发送出去。

接收数据则是相反的操作

OSI 可以映射到 TCP/IP，但这期间一、五、六层消失了

HTTP 利用 TCP/IP 协议栈逐层打包再拆包，实现了数据传输，但下面的细节并不可见

凡是由操作系统负责处理的就是四层或四层以下

凡是需要由应用程序（也就是你自己写代码）负责处理的就是七层。

你能用自己的话解释一下“二层转发”“三层路由”吗？

数据链路层–查找到目标MAC地址–数据转发

网络层–解析数据包头信息，找到目标IP地址–转发数据

你认为上一讲中的 DNS 协议位于哪一层呢？

域名解析 应用层

你认为 CDN 工作在那一层呢？

应用层
